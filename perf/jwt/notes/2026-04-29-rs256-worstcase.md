# JWT Module — RS256 Worst-Case Performance Analysis

## Environment

- **Module**: jwt-nginx-module
- **Build**: ReleaseSmall (`zig build -Doptimize=ReleaseSmall`)
- **Machine**: Intel Core i7 860 @ 2.80GHz, 8 cores, 16GB RAM, Arch Linux 6.19.12
- **Commit**: `2bba641`
- **Scenario**: RS256 (RSA-2048) via `jwt_key_file` keyval — compute-heavy worst case

## Motivation

The existing baseline covers only HS256 (inline HMAC secret). HS256 is the fastest path:
symmetric HMAC-SHA256 in OpenSSL assembly. RS256 is the production worst case: asymmetric
RSA-2048 signature verification via `EVP_DigestVerify`, which performs modular exponentiation
using OpenSSL's BIGNUM routines. This is orders of magnitude more expensive.

## Analysis

### 1. Throughput comparison — RSA is 2.1–2.4× slower

| Concurrency | HS256 rps | RS256 rps | Ratio (HS/RS) |
|------------|----------|----------|----------------|
| c=1        | 4,077    | 1,955    | 2.09×         |
| c=8        | 12,955   | 5,356    | 2.42×         |
| c=32       | 14,725   | 6,212    | 2.37×         |

At single-concurrency, RS256 achieves **1,955 rps** vs HS256's 4,077 — a 2.1× slowdown.
At saturation (c=32), RS256 plateaus at **6,212 rps** vs HS256's 14,725 — a 2.4× slowdown.

The gap widens at higher concurrency because RSA operations consume more CPU per request,
leaving less headroom for the nginx event loop to context-switch between concurrent connections.

### 2. Latency — RSA adds ~0.2ms per request

At c=1:
- **p50**: RS256 0.39ms vs HS256 0.17ms — +0.22ms RSA overhead
- **p95**: RS256 0.76ms vs HS256 0.45ms — +0.31ms
- **p99**: RS256 3.26ms vs HS256 1.13ms — +2.13ms (RSA tail latency)

The p50 overhead of 0.22ms is the core RSA-2048 signature verification cost.
This maps to ~220μs per RSA verify on a 2.8GHz Nehalem — consistent with OpenSSL's
`RSA_verify` performance on this class of hardware.

### 3. Perf-stat — why IPC is higher for RSA

| Metric              | HS256    | RS256     | Notes                          |
|---------------------|----------|-----------|--------------------------------|
| IPC                 | 0.74     | **1.11**  | RSA has higher ILP             |
| Instructions/req    | ~60K     | ~427K     | 7.1× more instructions         |
| Cycles/req          | ~81K     | ~385K     | 4.8× more cycles               |
| Branch miss rate    | 4.1%     | 4.2%      | Essentially identical           |
| Cache miss rate     | 0.27%    | 0.45%     | RSA key material is larger      |

**Key insight**: RS256 has **1.11 IPC** vs HS256's 0.74 — RSA is *more* instruction-level-parallel,
not less. OpenSSL's BIGNUM multiplication uses vectorized (SIMD) Montgomery multiplication on
x86_64, which can sustain >1 IPC because independent big-integer limbs are processed in parallel
with few data dependencies. HMAC-SHA256, by contrast, has a tight dependency chain in its
compression function, limiting IPC to ~0.74.

However, RS256 executes **7.1× more instructions per request** (427K vs 60K), so the absolute
cost is much higher despite the efficiency gain.

### 4. Cache behavior — RSA key material footprint

RS256 shows 0.45% cache miss rate vs HS256's 0.27%. The RSA public key (256-byte modulus +
exponent) and BIGNUM working buffers (several KB of stack) create a larger working set than
HMAC's 32-byte secret + 64-byte SHA-256 state. However, at 0.45%, the cache miss rate is
still excellent — the entire RSA hot path fits in L1/L2 cache.

### 5. Scaling behavior — RSA saturates earlier

| Concurrency | HS256 rps | RS256 rps | HS256 speedup | RS256 speedup |
|------------|----------|----------|---------------|---------------|
| c=1        | 4,077    | 1,955    | 1.0×          | 1.0×          |
| c=8        | 12,955   | 5,356    | 3.2×          | 2.7×          |
| c=32       | 14,725   | 6,212    | 3.6×          | 3.2×          |

RS256 scales slightly worse than HS256 under concurrency (3.2× vs 3.6× total speedup).
This is because each RSA request occupies the CPU for longer (0.39ms vs 0.17ms), reducing
the nginx event loop's ability to interleave concurrent request processing.

### 6. RSA vs HMAC cost breakdown (approximate)

Per-request cost at c=1:
- **nginx framework overhead**: ~0.10ms (from `reject-no-token` baseline: 6,301 rps → 0.16ms/req)
- **HS256 add-on**: ~0.07ms (HMAC-SHA256 + base64 decode)
- **RS256 add-on**: ~0.29ms (RSA-2048 verify + base64 decode + key lookup)

The RSA signature verification itself accounts for ~0.22ms of the RS256 add-on.
This is consistent with OpenSSL `RSA_verify` benchmarks on Nehalem-class hardware:
~4,500 RSA-2048 verifies/second/core.

### 7. Production implications

- **1,955 rps per core** for RS256 JWT validation is sufficient for most API gateway scenarios
- 8-core server at c=32 parallelism: **~50,000 rps** aggregate (8 × 6,212)
- Adding claim extraction would add ~13% overhead (from HS256 valid-claims data), bringing
  RS256 + claims to ~1,700 rps/core
- RSA-4096 would be ~4× slower than RSA-2048 (cubic scaling of modular exponentiation)
- ECDSA (ES256) would be ~1.5–2× faster than RSA-256 — a better choice for high-throughput

## Summary

RS256 (RSA-2048) is the production worst-case JWT verification path, delivering **1,955 rps**
at single-concurrency — 2.1× slower than HS256. The bottleneck is RSA modular exponentiation
(~427K instructions/request), not memory or I/O. For high-throughput JWT validation, prefer
HMAC (shared secrets) or ECDSA (smaller signatures, faster verification).
