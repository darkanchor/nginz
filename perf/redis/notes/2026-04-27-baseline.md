# Redis Module — Baseline

## Setup

- Module: `redis-nginx-module` (nginx upstream proxy to Redis)
- Build: ReleaseSmall
- Backend: Redis mock (`tests/mocks/redis.js`) on localhost:16379
- Machine: Intel Core i7 860 @ 2.80GHz, 8 cores, 16GB RAM, Arch Linux 6.19.12

## Command

```bash
bun perf/redis/benchmark/run.js --concurrency=1,8,32 --requests=200 --warmup=20
```

## Artifact

`perf/redis/benchmark/output/2026-04-27T15-39-44.408Z-redis-releasesmall/`

## Analysis

### Architecture advantage
The redis module uses nginx's built-in upstream mechanism — a mature, heavily
optimized non-blocking I/O path. At c=1, it achieves 2066 rps with p50=0.39ms
against a local mock.  This is ~5.5x faster than pgrest's pooled PostgreSQL
path at equivalent concurrency.

### Payload sensitivity
From 30 B to 8 KB, throughput drops only 24% (2066→1571 rps at c=1).  The
nginx upstream buffer chain handles response body copies efficiently, and the
Redis RESP parser is lightweight.

### Scaling limit
Maximum throughput is ~4800 rps at c=32, regardless of payload size.  This is
the single-core limit of the nginx worker process — 0.2ms per request on
average (4800 × 0.2ms ≈ 960ms of CPU per second).  The remaining 40ms/second
is event loop overhead.

### URI vs static key
No measurable difference — the URI-based key extraction is a few pointer
arithmetic operations in compiled Zig, not a bottleneck.

### Tail latency
At c=8, p99=8.8ms for small-read.  At c=32, p99=13ms.  These tails are from
Bun client-side scheduling (32 concurrent fetch workers in a single-threaded JS
runtime), not from nginz.  The nginx worker does zero context switches — this
pattern matches pgrest's perf-stat finding.
